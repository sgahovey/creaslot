# Fiche de résolution — US-12.1 Traçabilité des modifications de comptes par trigger et procédure stockée

Itération 12 (thème *SQL avancé / BDD*). Objectif pédagogique CDA, à la demande du
formateur : démontrer la maîtrise du **SQL avancé** (trigger + procédure stockée) en
complément de l'ORM Doctrine. Cas d'usage : traçabilité automatique des modifications
de comptes utilisateur **au niveau base de données**, complémentaire au journal
applicatif PHP existant (`journal_admin`, inchangé) — la trace est garantie même lors
d'une modification SQL directe hors application.

## Réalisation
- **Migration `Version20260629120000`** — trois objets en un seul `up()` :
  1. table technique `historique_utilisateur` (`id`, `utilisateur_id`, `champ_modifie`, `ancienne_valeur`, `nouvelle_valeur`, `date_modification` ; index `(utilisateur_id, date_modification)` ; **pas d'entité ORM** ; **pas de FK** pour survie RGPD) ;
  2. trigger `trg_historique_utilisateur` `AFTER UPDATE ON utilisateur`, traçant `email` / `nom` / `prenom` / `role` / `est_actif` / `mot_de_passe` ;
  3. procédure stockée `consulter_historique_utilisateur(IN p_utilisateur_id INT)`.
- **Sécurité** : le hash de mot de passe n'est **jamais** stocké (trace `'modifie'` → `'modifie'`). `derniere_connexion` exclue du trigger (bruit à chaque login).
- **Tests** : `tests/Integration/HistoriqueUtilisateurTriggerTest.php` (3 tests : le trigger remplit l'historique sur changement de nom ; non-fuite du hash ; la procédure retourne l'historique). **306 tests verts.**

## Difficultés techniques rencontrées et résolues

### 1. `DELIMITER` interdit dans une migration Doctrine
- **Contexte / problème** : un trigger et une procédure ont un corps multi-instructions ; en SQL interactif on change le délimiteur (`DELIMITER $$ … END$$`) pour que le `;` interne ne coupe pas le `CREATE`.
- **Diagnostic (cause racine)** : `DELIMITER` est une directive **cliente** (client `mysql`, phpMyAdmin), inconnue du serveur et de PDO ; la passer à `addSql()` provoque une erreur de syntaxe.
- **Solution retenue** : un seul `addSql()` par objet, corps `BEGIN … END` complet **sans** `DELIMITER` ni `;` final. Le driver PDO MySQL envoie une instruction par appel et n'essaie pas de découper la chaîne sur les `;` → le serveur reçoit le `CREATE TRIGGER … BEGIN … END` intégral et le parse correctement.
- **Alternatives écartées** : conserver `DELIMITER` (erreur de syntaxe PDO) ; multiples `addSql()` découpant le corps (le `CREATE TRIGGER` n'est pas fractionnable).
- **Résultat / vérification** : migration exécutée, trigger et procédure créés ; tests d'intégration verts.
- **Critère CDA touché** : composants d'accès aux données SQL ; démarche de résolution de problème.

### 2. Erreur MySQL 1419 (privilège SUPER / binary logging)
- **Contexte / problème** : la création du trigger échouait avec l'erreur **1419** (« You do not have the SUPER privilege and binary logging is enabled »).
- **Diagnostic (cause racine)** : avec le *binary logging* actif, la création d'un trigger par un utilisateur non-`SUPER` exige `log_bin_trust_function_creators=1`. L'utilisateur applicatif n'a pas le privilège `SUPER`, donc un `SET GLOBAL` au runtime est lui aussi refusé.
- **Solution retenue** : ajouter `command: --log-bin-trust-function-creators=1` au service `db` de `docker-compose.yml`.
- **Alternatives écartées** : `SET GLOBAL log_bin_trust_function_creators=1` au runtime (refusé, user sans `SUPER`) ; accorder `SUPER` à l'utilisateur applicatif (sur-privilège, mauvaise pratique sécurité).
- **Résultat / vérification** : après recréation du conteneur `db`, le `CREATE TRIGGER` passe sans erreur 1419.
- **Critère CDA touché** : composants serveurs sécurisés ; DevOps ; démarche de résolution de problème.

### 3. DDL auto-commit : rejouer une migration partiellement appliquée
- **Contexte / problème** : après la première tentative échouée (erreur 1419 sur le trigger), rejouer la migration échouait sur « table already exists ».
- **Diagnostic (cause racine)** : en MySQL, le DDL est **auto-commit** ; le `CREATE TABLE historique_utilisateur` avait été committé malgré l'échec ultérieur du trigger, alors que la migration n'était pas marquée comme exécutée → rejeu sur un état incohérent.
- **Solution retenue** : nettoyer via `DROP … IF EXISTS` des trois objets pour repartir d'un état propre, puis rejouer la migration.
- **Alternatives écartées** : supprimer manuellement la seule table (laisse un état partiel si d'autres objets existent) ; forcer la version dans `doctrine_migration_versions` (masque l'incohérence réelle du schéma).
- **Résultat / vérification** : migration rejouée intégralement depuis un schéma propre, les trois objets présents.
- **Critère CDA touché** : base de données relationnelle ; démarche de résolution de problème.

### 4. `CALL` de procédure sous PDO : curseur non libéré
- **Contexte / problème** : dans le test, l'appel `CALL consulter_historique_utilisateur(?)` faisait échouer la requête suivante avec « Cannot execute queries while other unbuffered queries are active ».
- **Diagnostic (cause racine)** : un `CALL` renvoie **plusieurs jeux de résultats** (le `SELECT` + un paquet de statut) ; sans libération du curseur, la connexion reste occupée.
- **Solution retenue** : dans le test, `fetchAllAssociative()` puis `free()` sur le `Result` DBAL avant toute requête suivante / le rollback.
- **Alternatives écartées** : enchaîner les `nextRowset()` manuellement (verbeux et fragile) ; ouvrir une seconde connexion (casse l'isolement transactionnel du test).
- **Résultat / vérification** : le test appelle la procédure puis poursuit (rollback) sans erreur ; assertion de l'historique verte.
- **Critère CDA touché** : tests d'intégration réalisés ; démarche de résolution de problème.

## Point de déploiement critique
`log_bin_trust_function_creators=1` est présent en **DEV** (`docker-compose.yml`, fait), mais doit **aussi** être présent en **PRÉPROD et PROD** (`compose.prod.yml` / configuration MySQL du VPS OVH). Sans ce paramètre, la migration `Version20260629120000` échouera au déploiement avec l'**erreur 1419**. À vérifier / ajouter **avant** la promotion `develop → preprod` (voir note ajoutée au `docs/runbook-deploiement.md`).

## Décisions assumées
- **Pas de FK** vers `utilisateur` : l'historique survit à la suppression du compte (même logique RGPD que `journal_admin`).
- **Pas d'entité Doctrine** : table technique alimentée par le trigger, lue via la procédure stockée.
- **Acteur applicatif non tracé par le trigger** (limite assumée) : le trigger ne connaît que `CURRENT_USER()` MySQL, pas l'utilisateur applicatif à l'origine de la modification ; cet acteur reste tracé par `journal_admin`. Les deux mécanismes sont **complémentaires** : `journal_admin` (qui a agi, côté application) et `historique_utilisateur` (quel champ a changé, garanti côté base).
